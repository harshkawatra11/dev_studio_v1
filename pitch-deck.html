▍ 02  ·  THE PROBLEM
Adding one feature
breaks four things at once.
Every time a developer ships a new feature in a production codebase, they manually update four separate layers — the database schema, the ORM model, the API routes, and the UI component. These layers are tightly coupled, but live in different languages, paradigms, and files. Miss one, the feature breaks. Update them in the wrong order, the build fails.
PAIN · 01
4 layers touched per feature — manually, every time.
PAIN · 02
Each layer has no awareness of what the others generated.
PAIN · 03
One wrong column name cascades across the entire stack.
The cost of a single missed layer is a broken production build.
▍ 03  ·  THE SOLUTION & USER JOURNEY
One sentence. Four specialist agents.
A working feature.
Workbench Studio v1.2 deploys a coordinated four-agent pipeline — one agent per layer, each an expert in its own domain. Each agent passes its output as context to the next, creating a chain where every layer knows exactly what was built above it.
▍ THE DIFFERENTIATOR
CHAIN
OF CONTEXT
Each agent sees the exact code its upstream agent produced.
No guessing. No hallucinated column names. No wrong function calls.

▍ 04  ·  TOOLS & TECH STACK
Built on a precise,
purpose-chosen stack.
COLUMN A
AI & Backend Layer
01
AI Engine — OpenAI Codex CLI
Spawned as four independent child processes, one per agent layer.
02
Agent Backend — Node.js 20 + Express 4
Orchestrates the pipeline; streams events via SSE.
03
Database — better-sqlite3
Synchronous, file-based, zero setup.
04
Real-time — Server-Sent Events
Native browser EventSource — no Socket.io overhead.
COLUMN B
Frontend & Tooling Layer
01
Studio UI — React 18 + Vite 5
Single-page app with fast hot module replacement.
02
Code Display — Monaco Editor
VS Code-quality diff viewer with before / after views.
03
Styling — TailwindCSS 3
Utility-first dark studio interface.
04
Multi-process — concurrently + nodemon
One command starts all four servers.
ARCHITECTURE
NOTE
!
The multi-agent architecture is not cosmetic. Each of the four Codex CLI processes runs independently, reads the sandbox codebase, and produces exactly one file. Downstream agents receive upstream output as inline context — not a file reference.
